# Why `sdks/neon/` is committed

Short answer: `@pulumi/neon` isn't a package on npm. It's generated locally from the upstream Terraform provider via `pulumi package add terraform-provider kislerdm/neon`, and we commit the generated output so `npm ci` works without an extra build step.

## Background

The Neon Pulumi provider is a community project bridged from `kislerdm/terraform-provider-neon` through `pulumi-resource-terraform-provider`. There's no published `@pulumi/neon` package on npm — the npm name is reserved by Pulumi but only ever resolves to a `file:` link to a locally generated SDK.

The generation flow is:

```
pulumi package add terraform-provider kislerdm/neon
  └─ downloads the bridge plugin (pulumi-resource-terraform-provider)
  └─ pulls the kislerdm/neon Terraform provider from a registry
  └─ introspects its schema
  └─ codegens TypeScript bindings into sdks/neon/
  └─ adds "@pulumi/neon": "file:sdks/neon" to package.json
  └─ adds the linked entry to package-lock.json
```

## Why commit it instead of regenerating per-install

The alternatives we considered:

1. **`.gitignore sdks/`, regenerate in CI before `pulumi up`.** Works, but every CI run pays the bridge plugin download (~30 MB) plus a registry round-trip. Local devs hit a "missing module" error after every `npm ci` until they remember to run `pulumi package add` themselves. The dirty working tree (post-generation modifications to `package.json` / `package-lock.json`) confuses `git status`.

2. **`preinstall` script that calls `pulumi package add` if `sdks/neon/` is missing.** Cleaner than (1), but requires `pulumi` on PATH before `npm ci`, and the first install is still chicken-and-egg with the lockfile.

3. **Commit `sdks/neon/`** (current choice). `npm ci` just works everywhere with no extra step. Cost: ~160 KB of generated TypeScript in git, regenerated only when bumping the provider version.

For an infra package that's rarely modified, (3) is the lowest-friction option.

## When to regenerate

Bump the Neon provider when you want a newer schema (new resources, new fields). From `infra/`:

```bash
pulumi package add terraform-provider kislerdm/neon
```

This will rewrite `sdks/neon/`, `package.json`, and `package-lock.json`. Commit the lot. Diff `sdks/neon/project.ts` (and friends) to see what changed in the upstream schema.

The pinned bridge version lives in `Pulumi.yaml` under `packages.neon.version`.

## Don't edit by hand

The contents of `sdks/neon/` are codegen output (every file starts with a `WARNING: this file was generated` banner). Local edits will be overwritten the next time someone runs `pulumi package add`. If you need to work around a provider bug, do it in the consumer (`infra/src/database.ts`) — not in the SDK.
